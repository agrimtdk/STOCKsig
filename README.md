## Project Overview

STOCKsig is a research-grade quantitative signal system built for Indian equity markets (NSE-listed large-cap stocks). It is not a price prediction engine. Instead, it classifies each stock's near-term outlook into one of three actionable categories:

- **BUY** — statistically favorable conditions for a long entry
- **HOLD** — insufficient evidence to act
- **SELL** — statistically unfavorable conditions

The system is designed around the principle that predicting *direction with actionable confidence* is more useful and measurable than predicting a raw price level. A raw price forecast gives no guidance on when to act, what threshold matters, or how much uncertainty exists. A calibrated signal with a confidence score gives a cleaner decision surface.

Signals are generated by combining:

- OHLCV price data (Open, High, Low, Close, Volume)
- Technical indicators computed from price history
- Rolling news sentiment extracted from financial headlines
- Earnings call transcript sentiment
- Macroeconomic indicators (CPI, USD/INR, interest rates)

The output is delivered through an interactive Streamlit dashboard that shows live signals, backtest analytics, feature importance rankings, model health diagnostics, and a portfolio replay tool.

## Problem Statement

Retail investors participating in Indian equity markets typically face a common set of challenges:

- **Information overload**: Dozens of news sources, analyst reports, and social commentary with conflicting conclusions.
- **Technical analysis complexity**: Indicators like MACD, RSI, and Bollinger Bands are individually well-documented but combining them systematically requires discipline.
- **Sentiment interpretation**: Financial news carries signal but separating noise from meaningful sentiment is manually intensive.
- **Risk-unaware execution**: Acting on a signal without accounting for volatility, drawdown risk, or position sizing leads to inconsistent outcomes.

STOCKsig attempts to address each of these by building a structured pipeline that ingests multi-source data, applies learned signal patterns, filters signals through technical confirmations, and reports confidence alongside each decision.

## Features

### Data Ingestion

- **Price data**: OHLCV history for NSE-listed equities via `yfinance` with configurable lookback windows.
- **News sentiment**: Articles collected from Finnhub and NewsAPI with Yahoo Finance as a fallback source. Raw articles stored in SQLite.
- **Transcript sentiment**: Earnings call transcript analysis using VADER sentiment and optional FinBERT (transformer-based) scoring.
- **Macroeconomic data**: CPI, USD/INR exchange rate, and RBI interest rate series fetched from FRED.

### Feature Engineering

Technical indicators computed per ticker using a grouped, look-ahead-safe pipeline:

| Feature | Description |
|---------|-------------|
| RSI (14) | Relative Strength Index — overbought/oversold momentum |
| MACD | Moving Average Convergence Divergence — trend momentum |
| MACD Signal / Histogram | MACD crossover indicators |
| Bollinger Band Width | Band compression as volatility proxy |
| ATR (14) | Average True Range — raw volatility |
| OBV | On-Balance Volume — volume-price relationship |
| SMA (20, 50) | Simple Moving Averages — trend baselines |
| EMA (12, 26) | Exponential Moving Averages — weighted trend |
| Volume Z-score | Standardized trading volume |
| Rolling Sentiment (3d) | Average sentiment, positive ratio, article count |
| Macro Indicators | CPI, USD/INR, Interest Rate |

All rolling features are shifted by one period to prevent data leakage into the training target.

### Machine Learning

- Multi-class classification: BUY (2), HOLD (1), SELL (0)
- Cross-validation using `TimeSeriesSplit` with 5 folds
- Models evaluated: Logistic Regression, XGBoost, LightGBM
- Class imbalance handled via `scale_pos_weight` and `class_weight` parameters
- Best model selected by macro-averaged F1 score
- Probability calibration using `CalibratedClassifierCV` (sigmoid method)

### Signal Engine

- Threshold-based signal activation: separate BUY and SELL probability thresholds
- Technical confirmation gates: RSI range checks, MACD direction, volume confirmation
- HOLD assigned when probability does not meet threshold or confirmation fails
- Confidence score reported alongside each signal

### Backtesting

- Execution on next trading day close (no lookahead)
- Position sizing: equal-weight per active position
- Maximum gross exposure cap enforced
- Transaction costs applied per trade
- Benchmark: NIFTY 50 equal-weight index reconstruction
- Roundtrip trade log with PnL and holding period

### Dashboard

- Live quote fetching with 15-minute cached refresh
- Watchlist with starred stocks and quick-scan panel
- Candlestick price chart with overlaid BUY/SELL signal markers
- Backtest equity curve vs benchmark
- Trade log with roundtrip analytics
- Feature importance rankings (XGBoost gain)
- Model calibration and probability distribution diagnostics
- Portfolio replay slider with position and trade history
- Chart image export (requires kaleido)

## Machine Learning Details

### Target Label Definition

Forward returns are computed over a 5-trading-day horizon:

```
BUY  = future_return > +2.0%
SELL = future_return < -2.0%
HOLD = otherwise
```

A 5-day horizon balances signal persistence (1-day is too noisy) with actionability (20-day is too slow for an equity signal system). The +/-2% threshold filters out micro-fluctuations while capturing directionally meaningful moves.

### Why TimeSeriesSplit

Standard random k-fold cross-validation would allow future data to appear in the training set, producing optimistically biased evaluation metrics. `TimeSeriesSplit` preserves temporal ordering: each fold trains only on data that preceded the validation window. This is the correct evaluation protocol for any time-series supervised learning problem.

### Class Imbalance

HOLD labels dominate the dataset (approximately 85-90% of all observations), as expected in real market data where strong directional moves are the exception. This is addressed by:

- Passing `class_weight` or `scale_pos_weight` arguments to model constructors
- Evaluating with macro-averaged F1 rather than accuracy
- Threshold tuning post-training to improve BUY/SELL recall

### Probability Calibration

Raw classifier probability outputs from gradient-boosted trees tend to be overconfident. `CalibratedClassifierCV` with sigmoid (Platt scaling) post-processing is applied to produce more reliable probability estimates, which are used directly in the confidence scoring system.

## Model Results

Best model: **XGBoost**

| Metric | Value |
|--------|-------|
| Mean Accuracy | 0.4324 |
| Mean Macro F1 | 0.3274 |
| SELL Recall | 0.2723 |
| HOLD Recall | 0.6208 |
| BUY Recall | 0.1445 |

**Interpretation**:

- **Accuracy of 43%** on a 3-class problem where random guessing yields 33% is a meaningful improvement, but it is modest. This is expected: equity markets are partially efficient and predictive signal is limited.
- **HOLD Recall of 62%**: The model correctly identifies most HOLD periods, which is the dominant class.
- **BUY/SELL Recall** is low, reflecting the difficulty of predicting rare directional events. This is common in financial classification problems and is why the signal engine applies additional confirmation filters.
- **Macro F1 of 0.33**: The model extracts real but limited predictive signal from the feature set.

**Limitations**:

- Signal quality degrades in structurally novel market conditions (e.g., post-COVID re-rating).
- SELL precision is low — the system should not be used in isolation to short stocks.
- The model does not account for liquidity, bid-ask spread, or market impact.
- Sentiment features are limited by article availability, which varies by stock.

## Signal Engine Logic

After calibrated probabilities are produced, the signal engine applies a two-stage filter:

**Stage 1 — Probability Threshold**

```
prob_buy  > BUY_THRESHOLD  → raw_signal = BUY
prob_sell > SELL_THRESHOLD → raw_signal = SELL
otherwise                  → raw_signal = HOLD
```

Default thresholds: BUY = 0.50, SELL = 0.55. These are tunable per deployment.

**Stage 2 — Technical Confirmation**

A BUY signal is confirmed only when supporting technical conditions are present (e.g., RSI not overbought, positive MACD direction, volume above average). A SELL signal requires inverse confirmation. If confirmation fails, the signal is downgraded to HOLD.

**Why confirmation matters**:

The model produces probability estimates but does not capture real-time momentum context at the moment of execution. Technical confirmations act as a sanity filter that prevents acting on stale or contradictory model outputs. A BUY signal with a falling MACD histogram and collapsing volume is a weaker signal than one with all three conditions aligned.

The confidence score reported on the dashboard reflects the raw model probability for the active class, providing a measure of the model's internal certainty.

**Plain-language interpretation**:

- **Total Return 134.28%** vs benchmark 53.48%: The strategy returned more than double the benchmark over the same period.
- **CAGR 19.01%**: Annualized compound growth rate, comparable to well-performing equity funds.
- **Sharpe 1.94**: Return per unit of total risk. Values above 1.0 are generally considered acceptable; above 1.5 is strong for an equity strategy.

- **Win Rate 80.36%**: 80% of completed trades closed at a profit.
- **56 trades over 5 years**: The system is not a high-frequency or overtrading system. It acts selectively.

**Important caveat**: Backtesting results are computed on historical data and are subject to overfitting risk. Past performance does not imply future results. The strategy has not been validated in a live trading environment.

## Installation

**Requirements**: Python 3.10 or higher.

```bash
# Clone the repository
git clone https://github.com/your-username/STOCKsig.git
cd STOCKsig

# Create and activate a virtual environment
python -m venv venv

# Windows
venv\Scripts\activate

# macOS / Linux
source venv/bin/activate

# Install dependencies
pip install -r requirements.txt
```

**Optional**: For PNG chart export support, install kaleido separately if not included in your environment:

```bash
pip install kaleido>=0.2.1
```

**API Keys** (optional):

Create a `.env` file or set environment variables for enhanced data coverage:

```
FINNHUB_API_KEY=your_key_here
NEWS_API_KEY=your_key_here
FRED_API_KEY=your_key_here
```

If keys are not provided, the system falls back to Yahoo Finance for news and uses default macro values.

## Running Locally

Each pipeline phase is run sequentially. Outputs from each phase are consumed by the next.

```bash
# Phase 1: Data Ingestion
# Downloads OHLCV data, news articles, transcripts, and macro series
python -m src.ingestion.run_phase1

# Phase 2: Feature Engineering
# Computes technical indicators, rolling sentiment, labels
python -m src.features.run_phase2

# Phase 3: Model Training
# Trains and evaluates models, saves best calibrated model
python -m src.models.run_phase3

# Phase 4: Signal Generation
# Applies thresholds and confirmation filters to produce signals.parquet
python -m src.signals.run_phase4

# Phase 5: Backtesting
# Simulates portfolio execution, generates equity curve and trade log
python -m src.backtest.run_phase5

# Dashboard
streamlit run app.py
```

**Run the test suite:**

```bash
pytest
```

## Deployment

STOCKsig is configured for deployment on [Streamlit Community Cloud](https://streamlit.io/cloud).

**Steps:**

1. Push the repository to GitHub. Ensure `data/demo/` is committed (it is whitelisted in `.gitignore`).
2. Go to [share.streamlit.io](https://share.streamlit.io) and connect your GitHub repository.
3. Set the main file path to `app.py`.
4. Deploy. The app will install dependencies from `requirements.txt` automatically.

**Notes:**

- The dashboard will use pre-computed data from `data/demo/` on cloud deployments where the full pipeline has not been run.
- Heavy pipeline dependencies (`torch`, `transformers`, `numba`, `vectorbt`) are commented out in `requirements.txt` to stay within cloud resource limits. These are only needed when running the full training pipeline locally.
- The `.streamlit/config.toml` file configures the theme and sets `headless = true` for server compatibility.

## Future Improvements

The following directions are identified for future development:

- **Richer alternative data**: Options open interest, FII/DII flow data, and insider transaction filings as additional signal sources.
- **Options flow integration**: Implied volatility surface and put/call ratio as sentiment proxies for large-cap stocks.
- **Social sentiment**: Structured extraction from financial forums, Reddit India investment communities, and Twitter/X firehose for retail sentiment tracking.
- **Intraday signals**: Extending the pipeline to support 15-minute or hourly bar data for shorter-horizon signal generation.
- **Reinforcement learning**: Replacing the threshold-based signal engine with a learned policy that directly optimizes a risk-adjusted return objective.
- **Portfolio optimization**: Adding mean-variance or risk-parity position sizing on top of the signal layer to replace equal-weight allocation.
- **Sector-level signals**: Aggregating stock-level signals into sector momentum overlays for top-down portfolio construction.
- **Live execution integration**: Connecting the signal engine to a broker API (e.g., Zerodha Kite Connect) for automated order placement with appropriate risk controls.
